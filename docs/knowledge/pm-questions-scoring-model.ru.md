# Вопросы к PM: Модель оценивания

Связанные истории: [Эпик 01 -- Основа оценивания](../roadmap/01-assessment-foundation/_epic.ru.md)
Связанные файлы: `src/types/state.ts`, `src/types/assessment.ts`

---

## Контекст

У нас есть три противоречивых сигнала о том, как оценивать отдельные критерии в Оценке этапов развития. Этот документ предоставляет полный контекст для формулирования правильного вопроса к PM.

## Три модели

### 1. Стандартное соглашение VB-MAPP

VB-MAPP (Verbal Behavior Milestones Assessment and Placement Program) доктора Марка Сандберга использует **шкалу с половинными баллами на один этап**:

| Балл | Значение                                   |
| ---- | ------------------------------------------ |
| 0    | Навык отсутствует                          |
| 0.5  | Навык формируется или частично проявляется |
| 1    | Навык полностью проявлен / освоен          |

Каждый этап внутри уровня вносит 0, 0.5 или 1 в общий балл уровня. Уровень (например, Уровень 1) содержит 170 этапов по всем областям навыков, поэтому максимальный балл составляет 170. Такая детализация позволяет специалистам отслеживать частичный прогресс и различать "никогда не наблюдалось" и "иногда наблюдалось".

### 2. Исходный документ оценки (предоставлен экспертом по предметной области)

Указано: "Каждый критерий оценивается от 1 до 4. Вычисляется итог по секции."

Это предполагает **4-балльную шкалу типа Лайкерта**, вероятно соответствующую уровням владения:

| Балл | Возможное значение                 |
| ---- | ---------------------------------- |
| 1    | Не продемонстрирован               |
| 2    | Формируется с подсказками          |
| 3    | Продемонстрирован с поддержкой     |
| 4    | Полностью самостоятельный / освоен |

Это отступление от стандартного оценивания VB-MAPP, но предлагает большую детализацию, чем шкала с половинными баллами для отдельных критериев.

### 3. Техническая спецификация MVP

Указано: бинарная оценка **Да/Нет** на критерий.

| Ответ | Значение                                         |
| ----- | ------------------------------------------------ |
| Да    | Навык продемонстрирован -- отметить, продолжить  |
| Нет   | Навык не продемонстрирован -- перейти к обучению |

Это простейшая модель. Она согласуется с принципом UX "простые решения" и минимизирует когнитивную нагрузку для родителей.

## Соображения

### Аргументы за бинарную модель (Да/Нет)

- Минимальная когнитивная нагрузка для родителей (не специалистов)
- Чёткое следующее действие: Да = готово, Нет = обучение
- Исключает субъективную интерпретацию промежуточных баллов
- Быстрее всего реализовать
- Соответствует принципу "обучение через действие"

### Аргументы за 3-балльную модель (стиль VB-MAPP: 0/0.5/1)

- Отраслевой стандарт -- совместим с клинической отчётностью
- Фиксирует "формирующиеся" навыки, что критически важно для отслеживания прогресса во времени
- Ребёнок с оценкой 0.5 по многим критериям выглядит совсем иначе, чем ребёнок с 0 -- бинарная модель теряет этот сигнал
- Специалисты или терапевты, просматривающие данные, ожидают этот формат

### Аргументы за 4-балльную шкалу

- Наиболее детальное отслеживание прогресса
- Различает выполнение с подсказками и самостоятельное выполнение
- Соответствует спецификации исходного эксперта по предметной области
- Требует больше объяснений для надёжного оценивания родителями

### Гибридный вариант

- Использовать **бинарный ввод (Да/Нет) как интерфейс для родителей** в MVP
- Внутреннее отображение: Да = 1, Нет = 0
- Позднее добавить 0.5 ("Иногда / Формируется") как третий вариант
- Это сохраняет простой UX сейчас, оставляя возможность для совместимости с VB-MAPP

## Вопросы к PM

1. **Кто является основной аудиторией для результатов оценки?** Только родители, или специалисты/терапевты тоже будут просматривать данные? Если специалистам нужна совместимость с отчётностью VB-MAPP, бинарной модели может быть недостаточно.

2. **Насколько важно отслеживание частичного прогресса?** Ребёнок, который "иногда" может попросить предметы (формирующийся навык), в сравнении с тем, кто никогда этого не делает -- нужно ли приложению различать эти ситуации? Если да, бинарная модель теряет критический сигнал.

3. **Является ли шкала 1--4 из исходного документа жёстким требованием эксперта по предметной области или это было иллюстративно?** Важно понять, было ли это осознанным клиническим решением или приблизительным наброском.

4. **Подойдёт ли поэтапный подход?** Бинарная модель для запуска MVP с запланированным переходом на 3-балльную (0/0.5/1) в ближайшей итерации. Это сохраняет простоту MVP, признавая, что оценке нужны нюансы.

5. **Должна ли модель данных поддерживать более детальное оценивание с первого дня, даже если UI показывает только Да/Нет?** Это предотвратит миграцию данных в будущем.

## Рекомендация (с точки зрения разработки)

Спроектировать модель данных для хранения числового балла (0, 0.5, 1) независимо от того, что показывает UI. Для MVP интерфейс показывает Да/Нет, что внутренне отображается в 1/0. Это даёт нам:

- Простой опыт для родителей сейчас
- Отсутствие миграции данных при добавлении варианта "формируется" позднее
- Путь к совместимости с VB-MAPP без переделки
